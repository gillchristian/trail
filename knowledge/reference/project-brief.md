# Project brief

> The single source of truth for *what this project is*. Everything in `planning/` derives from this. Update it explicitly when scope changes; don't let intent drift silently.

## Status

**Not yet written.** Waiting for the user to describe the project. Once received, capture verbatim where possible, then summarize.

## Sections to fill in (when brief arrives)

### What it is
One paragraph: what does this project do, for whom?

### Why it exists
The problem being solved or the value being created. The "north star."

### Hard constraints
Things that are non-negotiable: language, platform, performance, deadlines, integrations.

### Soft preferences
Things the user would prefer but could compromise on.

### Out of scope
Explicit list of things this project will *not* do. Prevents scope creep.

### Success criteria
How we know the project is "good enough to ship."

### Open questions
Things the brief doesn't answer yet. Each one is a candidate ADR or blocker.

### Raw notes
The user's words, captured as-is, before I summarized. Useful when I think my summary may have lost nuance.
